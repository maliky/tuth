  - app/shared/fuzzy_matching.py defines name_similarity, but it currently calls    sim_name_token, which is commented out—so calling this function will raise a    NameError. It also relies on Python’s difflib.SequenceMatcher for both
    surname and given-name scoring, with custom weighting/penalties.
  - app/finance/utils.py doesn’t have any name-similarity logic; it only has
    tuition helpers. The only other name-matching utilities live in app/shared/
    importing/username_matching.py (Jaro-Winkler similarity for usernames) and
    app/people/utils.py (name_distance, now order-insensitive and backed by
    rapidfuzz Jaro-Winkler).

  Given that, I prefer the Jaro-Winkler-based approach (as in people/utils.py or  shared/importing/username_matching.py) because:

  - It uses a well-tested, fast C-backed implementation (rapidfuzz) instead of
    Python-level SequenceMatcher.
  - It produces a normalized distance/similarity, is symmetric, and—after
    canonicalization/sorting tokens—handles name reordering more gracefully.
  - It doesn’t depend on a missing helper (sim_name_token), so it won’t throw.

  If you want to improve scalability for ~10k×10k name comparisons (100M pairs),  avoid Python loops and leverage vectorized/compiled distance routines:

  - Use rapidfuzz.distance.JaroWinkler.cdist(left_list, right_list, workers=-1)
    (or process.cdist) to compute a full similarity matrix in C with
    multithreading. You can then threshold or keep top-k per row without holding    the full matrix by streaming/chunking.
  - Pre-canonicalize and lowercase all names once; optionally sort tokens to
    make comparisons order-insensitive as in people/utils.py.
  - If 100M pairs is still too large to hold, block on simple keys (e.g., first
    letter(s) of last name) to reduce candidate sets, then run cdist on each
    block.

  So: fix shared/fuzzy_matching.name_similarity by reinstating or replacing
  sim_name_token (or swap to rapidfuzz), and for large-scale matching, use
  rapidfuzz’s cdist/top-k utilities rather than Python loops.
