** Scope and method
This first Django admin audit used headless Chromium/Selenium against preprod, behind basic auth =tusis:tusis=, with the Django admin test user =dev=.

The pass was intentionally non-destructive.
It loaded admin pages and GET-only admin endpoints but did not submit saves, deletes, admin actions, imports, exports, or mutating forms.

** Executive state
The Django admin shell itself is reachable and most admin pages load.
The =dev= admin user authenticated successfully, the dashboard rendered, and all 44 discovered model changelists were reachable.

The main risk is not basic access.
The main risk is deeper admin behavior: search/facet combinations, history views on admins without a compatible history manager, and a few high-cost admin pages.

The priority is to fix the 500s before using admin as a dependable MWE fallback for staff operations.
The second priority is to reduce the slow pages that make admin feel unreliable under operational use.

** What loaded correctly
- =/admin/= loaded after basic auth and Django login.
- All 44 model changelists discovered from the admin index returned a usable admin page.
- Most add-form GETs loaded without submitting data.
- Most autocomplete endpoints returned JSON successfully.
- Most change and history views loaded when an existing row was available.
- Import/export object-tool GETs did not surface as a primary failure in this pass.

Skipped deeper checks:
- Some models had no existing object link on the first changelist page, so change/history GETs were skipped for those models.
- Examples include finance invoice/payment/scholarship models, transcript requests, impersonation logs, and timetable terms.
- These skips are data-state skips, not necessarily application defects.

** Blocking findings
*** DONE P0 - History views return 500 for admins without a compatible history manager
Affected admin paths:
- =/admin/academics/curristdenroll/<id>/history/=
- =/admin/people/faculty/<id>/history/=

Observed behavior:
- Both history pages returned HTTP 500.
- Server logs show =AttributeError: 'Options' object has no attribute 'simple_history_manager_attribute'= from =simple_history.admin=.

Likely source areas:
- =app/academics/admin/core.py=
- =app/people/admin/core.py=
- =app/shared/admin/mixins.py=

Interpretation:
Some admin classes inherit a history-capable admin base, but their model does not appear to expose the corresponding simple-history manager.
The admin history link therefore exists but crashes when used.

Recommended fix:
Use a non-history admin base for models without simple-history tracking, or add/verify historical tracking for those models before exposing history views.

Resolution:
Faculty and curriculum-enrollment admin now use a college-restricted admin base that does not expose SimpleHistory views.
This keeps the admin usable without adding model history migrations.

Verification:
Focused Django admin GET regression tests passed for faculty history and curriculum-enrollment history.

*** DONE P0 - Department-backed search/autocomplete can return 500
Affected admin paths:
- =/admin/academics/department/?q=zzzz-admin-audit-no-match=
- =/admin/autocomplete/?app_label=people&model_name=roleassignment&field_name=department&term=a=

Observed behavior:
- Department admin search returned HTTP 500.
- Role assignment department autocomplete returned HTTP 500.
- Server logs show =FieldError: Unsupported lookup 'icontains' for ForeignKey or join on the field not permitted=.

Likely source areas:
- =app/academics/admin/core.py=
- =app/people/admin/core.py=

Interpretation:
=DepartmentAdmin.search_fields= includes a ForeignKey field directly, which Django cannot search with the default =icontains= lookup.
Any autocomplete that relies on DepartmentAdmin search can inherit the same failure.

Recommended fix:
Replace direct ForeignKey search entries with concrete related text fields, such as college code/name fields.
Add a focused admin search/autocomplete test for department and role assignment department autocomplete.

Resolution:
Department admin search now targets concrete department and college text fields instead of the direct college ForeignKey.
Role-assignment department autocomplete now inherits a safe department search configuration.

Verification:
Focused Django admin GET regression tests passed for department search and role-assignment department autocomplete.

*** DONE P0 - Finance facet/filter links can return 500
Affected admin paths:
- =/admin/finance/crsinvoice/?_facets=True&q=zzzz-admin-audit-no-match=
- =/admin/finance/stdsemesterinvoice/?_facets=True&q=zzzz-admin-audit-no-match=

Observed behavior:
- Both finance facet/filter GETs returned HTTP 500.
- Server logs show =FieldError= stating a field cannot be both deferred and traversed with =select_related=.

Likely source area:
- =app/finance/admin/core.py=

Interpretation:
Finance admin query optimization conflicts with Django admin facet/filter rendering for these query combinations.
This is especially risky because the broken links are generated by the admin filter UI itself.

Recommended fix:
Review =list_select_related= and any queryset deferral/only/defer behavior used by the finance admin and scoped filters.
Keep the selected related fields consistent with the fields rendered by list display and filters.
Add a regression test for finance invoice changelist with =_facets=True= and a search query.

Resolution:
The amount-due filter now computes thresholds from value rows instead of iterating over =.only()= model instances.
This avoids the deferred-field and =select_related= conflict when Django admin renders facet/filter pages.

Verification:
Focused Django admin GET regression tests passed for course-invoice and student-semester-invoice facet/search combinations.
