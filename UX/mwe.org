* MWE Operator Manual
This section is a practical manual for demonstrating the app with the runtime MWE accounts above. It describes the current user experience and names admin-backed gaps explicitly, so the demo does not promise a portal screen that is not yet implemented.

** Working terminology
- Enrollment desk :: staff who create or update the student record, assign the curriculum/program, and verify core profile information.
- Finance clerk :: human office role receiving payment evidence. In the app today, use the =Finance Officer= account for invoice/payment operations; use =Cashier= only when demonstrating a distinct payment-intake role.
- Registrar :: staff who review grades, transcripts, registration status, and official academic records.
- Registrar Officer :: registrar role with operational authority to open or close semester/course windows.
- Academic / Faculty :: instructor who teaches sections and submits grades. The current portal exposes an instruction dashboard, while grade entry itself remains admin-backed.
- Registration :: the student's course/section enrollment row. In MWE workflows this can be created by student course selection, imported legacy data, or registrar/admin action.
- Financial clearance :: invoice/payment state that lets the registration move from pending payment toward cleared/official status.
- Grade sheet / transcript / roster :: use "grade sheet" for a student's semester/course-grade report, "transcript" for the registrar official preview, and "roster" for a section list used by faculty/registrar.

** Full student lifecycle from first enrollment to re-registration
1. The student or candidate presents required information to the Enrollment desk.
2. Enrollment logs in as =test_enrollment= and opens =/staff/enrollment/=.
3. Enrollment uses =/students/new/= to create the student or to update a selected student record.
4. Enrollment records the student's identity, contact data, and curriculum/program assignment. If a richer edit is needed, Enrollment uses =/students/admin-edit/= to search the student and continue the profile work.
5. The student receives login credentials and opens =/auth/login/=, then is routed to =/student/dashboard/=.
6. The student reviews profile, current registration status, balance, and curriculum from the dashboard.
7. The student opens =/student/curriculum/= to understand the program courses and opens individual course details when prerequisites or descriptions are needed.
8. During an open registration period, the student selects available sections from the dashboard registration area. The system should show eligible courses, seats, credits, and invoice impact where data exists.
9. The created registrations start as pending payment until finance clears them.
10. The student opens =/student/invoice/statement/= to review charges and communicates with Finance for payment.
11. Finance records or clears payment. Once cleared, the student sees the balance and registration status update on the dashboard or statement.
12. Registrar validates the official academic record and uses the registrar grade/transcript tools when the semester is complete.
13. For the following semester, the same student logs in again, checks the new open/current semester, selects the next eligible sections, and repeats the payment and clearance cycle.

* <2025-06-27 Fri>

I'm going to give a demonstration of the Tuth software, the computer system for managing administration, or rather, managing student registration at our university. The Django base is already good, as you can see from your memory. Now, it's about working with a program like OpenAI's Codex to generate minimal working examples for the following cases. The goal is to restrict object access for different users, so I'd like it to suggest access restrictions for the registrar, who manages the databases containing student grades. I'd also like proposals for restrictions on staff involved in enrollment, meaning student registrations. These staff members create student profiles after verifying their documents, allowing the creation of new student objects. There is the head of the Enrollment department, who must be able to modify permissions for employees. I can already see several roles here: registrar, enrollment head, and enrollment staff. Of course, there's also the student role.

I also want to add a particular role related to finance or the finance department. That's the cashier, the person to whom students deposit money, who can send a message to the financial officer confirming that a particular student has deposited a certain amount of money for one of the student's registrations.

In this Minimal Working Example, a student comes to the enrollment office; we create their profile, assign their program, after which the student can view courses available in their program and register according to Business Rules that I'll present later.

There's a maximum number of credits that a student can take. This maximum number of credits must be modifiable by the VPA for the entire university. Additionally, each student has a per-semester credit limit that can be modified by the DEAN. Again, these are additional roles we need to identify.

For our example, we'll assume the maximum credit load is 12. Courses typically have 3 credits, so the student can select up to 4 courses during registration. What's important is to accurately calculate the number of credits chosen by a student.

Thus, depending on the number of credits—we'll call this the student's credit load at instant T—we can display additional courses from their program for which they can register. Registering for a course is equivalent to creating a new line in the student's registration table.

For this minimal working example, we'll assume we have two students. One is making their selections—they're both freshman students—and when the second student makes their selection, they actually see some courses as already full. That's another business logic we must implement: courses have a limited number of seats. Therefore, for this minimal working example, we'll have courses with only one seat left so we can see what happens when a course fills up.
