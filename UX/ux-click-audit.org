** Scope and method
The original audit used headless Chromium/Selenium against preprod with basic auth =tusis:tusis=.
It used the runtime MWE accounts documented in =UX/mwe.org=.

The crawl covered:
- public landing and login
- student and prospecting-student portals
- staff, faculty, chair, dean, VPAA, registrar, registrar officer, finance, finance officer, cashier, enrollment, enrollment officer, IT, and donor logins
- representative mobile pages for public, login, student, finance, registrar, enrollment, and dean

The P2/P3 follow-up verification used the same public preprod route with the runtime MWE accounts and stricter checks for 500, 403/access-denied pages, visible admin-first wording, and primary =/admin/= anchors.

The audit intentionally did not click Django admin or submit destructive/mutating forms.
Skipped mutating actions include student registration/drop, finance payment creation/update, semester status update, and student delete.

** Blocking findings
*** DONE P0 - Student/registrar primary links return 500
The click audit found repeated 500s on primary MWE links:
- =/student/curriculum/=
- =/student/invoice/statement/=
- =/student/payment/receipt/<semester>/=
- =/registrar/grades/=

Server logs showed =TemplateDoesNotExist= for old template names after the portal template rename/harmonization.
A static render-name check also found additional old template references that were not all reached by the crawl.

Affected source areas:
- =app/website/views/student_curriculum.py=
- =app/website/views/student_dashboard.py=
- =app/website/views/student_payment_receipts.py=
- =app/website/views/student_sections.py=
- =app/website/views/registrar.py=

Resolution:
Updated stale view render targets to the current portal template names.
Targeted Selenium verification passed for student curriculum, invoice statement, payment receipt, registrar grades, and registrar-officer grades.

** Primary workflow issues
*** DONE P1 - Role switcher exposes inaccessible inherited staff pages
The dean and VPAA role-switcher/sidebar experience exposed workspace options that could lead to 403 responses.
Observed examples included dean navigation to faculty and VPAA navigation to chair.

Resolution:
Role switcher visibility and staff dashboard permission now share the same transitive role-access closure.
Targeted Selenium verification passed for dean access to the faculty workspace and VPAA access to the chair workspace.
The P2/P3 strict browser run also confirmed representative backstage dashboards do not show access-denied pages, including the runtime IT group spelling =It=.

*** DONE P1 - Student dashboard overflows horizontally on mobile
The mobile smoke test found global horizontal overflow on the student dashboard.
The broad elements were the portal sidebar/content/header layout and student dashboard sections.

Resolution:
Adjusted the portal shell, sidebar navigation, KPI grid, and course-table containment for mobile widths.
Targeted Selenium verification passed with no global horizontal overflow on the student dashboard.

** UX harmonization issues
*** DONE P2 - Django admin remains too visible in staff workflows
Most staff pages previously exposed =Advanced admin= in the top-right header, and several role action cards linked directly to admin models.
This made the staff portal feel like a skin over Django admin rather than a complete Tusis workspace.

Resolution:
- Removed the staff-shell =Advanced admin= header action.
- Filtered Django admin destinations out of role action cards.
- Replaced finance, registrar, enrollment, and staff dashboard actions with portal-native task destinations.
- Updated the public landing copy to describe advanced tools as outside the primary role workflow.
- Added the runtime =It= group spelling to the IT workspace so IT resolves to the portal instead of an inaccessible page.

Verification:
The strict P2/P3 Selenium run passed for representative staff, dean, finance, cashier, enrollment officer, registrar, donor, and IT pages with no visible admin-first wording, no primary =/admin/= anchors, and no access-denied pages.

*** DONE P2 - Enrollment create/edit remains admin-first
Enrollment pages rendered in the portal shell, but create/edit copy and actions explicitly sent users to Django admin.
Observed surfaces:
- =/students/new/=
- =/students/admin-edit/=
- =/students/=

Resolution:
- Added a portal-native student intake form for create/update.
- Student directory links now open portal profiles.
- Student detail exposes an =Update profile= action that returns to the portal form.
- Student lookup opens the portal profile instead of redirecting to Django admin.

Verification:
The strict P2/P3 Selenium run passed through enrollment directory, profile, update profile, create student, and lookup pages without leaving Tusis.

*** DONE P2 - Finance console is visually aligned but data/default states need UX tightening
The finance console loaded and looked consistent with the portal shell.
However, the default audited state showed no invoices, an empty submit control in the invoice-payment form, and admin escape links still visible.

Resolution:
- Added guided empty states for invoice and payment tabs.
- Added =Show all invoices= and =Show all payments= filter-reset actions.
- Rendered invoice selection controls only when invoice rows exist.
- Kept finance dashboard actions pointed to the portal console.

Verification:
The strict P2/P3 Selenium run passed the finance default invoices page, payments tab click, all-invoices GET filter, all-payments GET filter, and mobile finance smoke pages.

*** DONE P2 - Registrar course-window page is functional but operationally dense
The course-window page loaded, but it rendered a long list of per-semester POST forms with repeated =Update= buttons.
The audit skipped those mutating submits.

Resolution:
- Grouped semester windows by academic year.
- Kept each row operational but visually compact.
- Added a confirmation affordance before changing a semester window.

Verification:
The strict P2/P3 Selenium run passed desktop and mobile registrar course-window pages and confirmed grouped academic-year cards and confirmation-backed forms are present.

*** DONE P2 - Donor account falls through to generic staff dashboard
=test_donor= authenticated successfully but landed on the generic staff dashboard.

Resolution:
Added a donor role workspace with scholarship commitment metrics and beneficiary summary panels.

Verification:
The strict P2/P3 Selenium run confirmed =test_donor= lands on =Donor Sponsorship Portal= with =Scholarship commitments= visible.

** Lower-priority findings
*** DONE P3 - Some audit clicks were intentionally skipped or inconclusive
The original audit skipped POST forms by design.
Two finance GET-filter attempts produced stale-element Selenium errors during audit interaction, but the finance pages themselves loaded and rendered.

Resolution:
The P2/P3 follow-up reran the finance GET paths and tab click directly against preprod.
The audited finance default invoices, payments tab, all-invoices filter, and all-payments filter passed without stale-element failure.
Mutating POST actions remain intentionally out of this non-destructive UX audit.

*** DONE P3 - External/social/media imagery is not the current bottleneck
The site can be made functional without photography first.
The current goal is a comfortable MWE with a harmonized portal shell and working backstage workflows.

Resolution:
No photo dependency was introduced.
The landing page now leans on the Tubman/Tusis portal direction, consistent layout, typography, and task language rather than requiring university photography to carry the experience.
Images remain a polish task, not an MWE blocker.
