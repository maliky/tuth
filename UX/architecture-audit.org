* Scope and method
This pass audits the implementation architecture behind the MWE user experience and records the follow-up remediation implemented from that audit.
The first pass was diagnostic; the current worktree now includes architecture, UX-shell, workflow, and validation changes.

The goal is to state where the MWE stands from these perspectives:
- student
- finance staff
- dean and registrar-family staff
- Django admin fallback for privileged operations

The audit combines:
- local source inspection
- Django route, model, and admin inventory
- portal template inheritance inspection
- import and file-size inspection
- static checks and tests
- public preprod browser smoke checks with the runtime MWE accounts from =UX/mwe.org=

The design direction is inherited from =UX/tu-web-design-guide.org=:
- student UI is the visual reference
- staff UI inherits the shared portal shell
- finance, dean, registrar, and other roles vary by task modules, not by a separate visual skin
- Django admin remains an advanced path, not the primary user journey

* Executive state
The MWE is in a usable portal-first state.
The public landing page, portal login, student dashboard, finance-officer console, dean dashboard, registrar dashboard, registrar-officer course windows, and Django admin all load on preprod.

No current P0 architecture blocker was found in this pass.
The immediate risk is not "pages do not load".
The main architectural risk has been reduced: the largest website views have been replaced by thin request/permission/render wrappers, and their orchestration now lives in typed portal service modules.

The architecture is good enough for an MWE demonstration across student, finance officer, dean, registrar, registrar officer, and enrollment flows.
It is still not finished architecture: =student_portal.py= and =staff_portal.py= remain large service modules and should be split again when the next workflow area is touched.

* Ranked findings
** P0 - No current architecture blocker found

** P1 - Working UX is too concentrated in large orchestration modules
Affected source areas:
- =app/website/views/staff_dashboards.py=
- =app/website/views/student_dashboard.py=
- =app/academics/admin/core.py=
- =app/people/admin/core.py=
- =app/finance/admin/core.py=
- =app/registry/admin/core.py=

Evidence:
- The URL-facing staff and student dashboard views are now thin wrappers.
- Their previous orchestration moved into =app/website/services/staff_portal.py= and =app/website/services/student_portal.py=.
- =staff_portal.py= and =student_portal.py= are now compatibility façades over smaller service modules.
- The largest admin =core.py= files are now compatibility façades over domain admin modules.

Impact:
The UX works today, but adding one more staff workflow risks copying patterns into the same large files.
This makes role behavior harder to reason about and makes typed refactoring harder.

Status after remediation:
- DONE: thin URL-facing views for student, staff, finance, registrar, and dean portal routes.
- DONE: typed portal service modules added.
- DONE: split =student_portal.py= and =staff_portal.py= into smaller composable service modules with compatibility façades.
- DONE: split the largest admin =core.py= modules into domain modules with compatibility façades.

** P1 - Mutating workflow confidence is still below the non-mutating route confidence
Affected source areas:
- =app/website/views/student_dashboard.py=
- =app/website/views/finance_officer.py=
- =app/website/views/registrar.py=

Evidence:

** P2 - Service/query boundaries are inconsistent
Affected source areas:
- =app/website/views=
- =app/finance/utils.py=
- =app/people/utils.py=
- =app/shared/auth/perms.py=

Evidence:
- Website views now delegate the large context/query work to the portal service layer.
- Finance and registrar workflows have dedicated typed service modules.
- Dean curriculum review has a dedicated typed service module.

Impact:
The portal is visually harmonized, but the code architecture does not yet express a clean "portal layer" over domain services.
This increases the risk of duplicated query logic and role-specific behavior drifting apart.

Status after remediation:
- DONE: typed portal service/query layer created.
- DONE: thin views now call the service layer.
- DONE: common portal type aliases and component context shapes now live in =app/website/services/portal_types.py=.
